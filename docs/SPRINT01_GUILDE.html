2. Tư duy cấu trúc code Core ▾

Tách theo trách nhiệm, không viết tất cả trong một file:

Spec file → mô tả test scenario Page Object → mô tả UI element và action Test Data → dữ liệu đầu vào / expected result Config → môi trường chạy test

Spec chỉ gọi method, không được chứa locator:

// ✓ Đúng — spec gọi method await adminPage.searchByUsername(data.username); // ✗ Sai — locator nằm thẳng trong spec await page.locator('.oxd-input').nth(1).fill(data.username);
3. Page Object Model (POM) Core ▾

POM dùng để: gom locator, gom action về một nơi, giúp spec dễ đọc, sửa UI không ảnh hưởng nhiều test.

Thứ tự đọc POM:

constructor() → locator → action method → module.exports
class AdminPage { constructor(page) { this.page = page; this.usernameInput = page.locator('.oxd-input').nth(1); this.searchButton = page.getByRole('button', { name: 'Search' }); } async searchByUsername(username) { await this.usernameInput.fill(username); await this.searchButton.click(); } }

Locator = "đồ vật"  |  Method = "hành động"  |  Spec = "kịch bản"

4. 5 bước viết function trong POM ▾
BướcLàm gìVí dụ
1Xác định mục đíchSearch user theo username
2Xác định element cần thao tácusername input, search button
3Tạo locator trong constructorthis.usernameInput = ...
4Viết method hành độngasync searchByUsername()
5Gọi method trong specawait adminPage.searchByUsername(...)

Checklist khi viết function:

□ Function làm đúng 1 việc chính không? □ Tên function có dễ hiểu không? □ Locator có nằm trong POM không? □ Spec có bị dính locator không? □ Function có tái sử dụng được không?
5. Tư duy viết test case Core ▾

Mọi test case cần đủ 3 phần: Setup → Action → Assertion

① Data từ đâu ra?

Data không bao giờ được hard-code trong spec. Nó đến từ file JSON riêng, rồi được import vào spec.

// test-data/loginData.json { "username": "Admin", "password": "admin123" } // tests/Login.spec.js const data = require('../test-data/loginData.json'); // └── data.username = "Admin" // └── data.password = "admin123" // Dùng trong test: await loginPage.performLogin(data.username, data.password); // └── KHÔNG viết: performLogin('Admin', 'admin123') // └── Vì nếu password đổi → chỉ sửa JSON, không sửa code

② Locator gọi thế nào?

Spec không tự tìm element. Spec gọi method từ POM. POM mới là nơi chứa locator.

// ✗ SAI — spec tự gọi locator await page.locator('.oxd-input').nth(1).fill(data.username); await page.getByRole('button', { name: 'Login' }).click(); // ✓ ĐÚNG — spec gọi method từ POM await loginPage.performLogin(data.username, data.password); // └── Bên trong POM mới có: // this.usernameInput.fill(username) // this.loginButton.click()

③ Function chạy qua bao nhiêu lớp?

3 lớp: Spec gọi POM → POM dùng Locator → Locator tương tác Browser.

Lớp 1 — SPEC (kịch bản) await loginPage.performLogin(data.username, data.password) └── gọi method trong POM Lớp 2 — POM (hành động) async performLogin(username, password) { await this.usernameInput.fill(username) ← dùng locator await this.loginButton.click() ← dùng locator } └── dùng locator để tương tác UI Lớp 3 — BROWSER (thực thi) Playwright tìm element trên màn hình Playwright gõ text / click button └── đây là nơi action thật sự xảy ra
⚠️ Spec không biết browser. POM không biết data. Browser không biết kịch bản. Mỗi lớp chỉ làm đúng 1 việc.

④ Tại sao phải tách POM?

// Nếu KHÔNG dùng POM — locator nằm khắp nơi // Login.spec.js await page.locator('.oxd-input').nth(0).fill('Admin'); // Admin.spec.js await page.locator('.oxd-input').nth(0).fill('Admin'); // Dashboard.spec.js await page.locator('.oxd-input').nth(0).fill('Admin'); // → UI đổi class từ .oxd-input sang .hrm-input // → Phải sửa tất cả 3 file ↑ // Nếu DÙNG POM — locator chỉ có ở 1 nơi // LoginPage.js this.usernameInput = page.locator('.oxd-input').nth(0); // → UI đổi class → chỉ sửa LoginPage.js // → 3 spec file không cần đụng vào

⑤ Cách đọc code thực tế

Khi nhìn vào 1 dòng code trong spec, đọc từ ngoài vào trong theo luồng này:

// Dòng code: await adminPage.searchByUsername(data.username); // Đọc như sau: 1. "data.username" → Mở test-data/adminSearch.json → Tìm field "username" → giá trị là "Admin" 2. "adminPage.searchByUsername(...)" → Mở pages/AdminPage.js → Tìm method searchByUsername() → Bên trong: fill usernameInput → click searchButton 3. "usernameInput / searchButton" → Nhìn lên constructor của AdminPage → this.usernameInput = page.locator('.oxd-input').nth(1) → this.searchButton = page.getByRole('button', {name:'Search'}) 4. Kết quả trên browser: → Playwright gõ "Admin" vào ô input thứ 2 → Playwright click nút có text "Search"

✓ Test case hoàn chỉnh đọc đúng cách

test('[POSITIVE] TC_LOGIN_001 - Verify login with valid credentials', async ({ page }) => { // SETUP — chuẩn bị môi trường const loginPage = new LoginPage(page); // tạo POM instance const data = loginData.validUser; // lấy data từ JSON await loginPage.navigate(); // mở trang login // ACTION — thực hiện hành động await loginPage.performLogin(data.username, data.password); // └── POM fill username → fill password → click Login // ASSERTION — kiểm tra kết quả await expect(page).toHaveURL(/dashboard/); // └── verify URL chứa "dashboard" → login thành công });
6. Data Driven Testing ▾

Chỉ dùng data-driven khi đáp ứng đủ 3 điều kiện:

✓ Cùng flow ✓ Khác data ✓ Khác expected result

Không nên dùng khi:

✗ Flow khác nhau hoàn toàn ✗ Assertion khác quá nhiều ✗ Phải viết quá nhiều if/else ✗ Test trở nên khó đọc hơn

Bài học: Data-driven tốt khi làm test gọn hơn, tệ khi làm logic rối hơn.

7. Storage State Hay nhầm ▾

Vấn đề: Admin test cần login. Login test cần browser chưa login. Nếu dùng storageState global → Login test fail.

Giải pháp đúng:

setup project ↓ auth.setup.spec.js → tạo session → auth/user.json ↓ Admin.spec.js → dùng storageState ✓ Login.spec.js → browser sạch, KHÔNG dùng storageState ✓

Không phải mọi test đều cần cùng một browser state.

8. Tư duy debug Playwright ▾

Khi test fail, KHÔNG sửa locator ngay. Debug theo thứ tự:

1. URL hiện tại là gì? 2. Browser đang login hay chưa login? 3. storageState có được apply không? 4. Data input/expected có đúng không? 5. Locator có tìm đúng element không? 6. Assertion có đúng business rule không?

Fail message chỉ là triệu chứng. Root cause thường nằm ở flow trước đó.

9. Cách debug thực tế ▾
# Chạy riêng 1 file (xem file đó tự pass không) npx playwright test tests/Admin.spec.js --headed # Chạy toàn bộ suite (phát hiện lỗi config/session) npx playwright test # Mở report (xem fail ở step nào) npx playwright show-report

Khi fail, xem screenshot → video → trace để biết browser đang ở trang nào tại thời điểm fail.

10. Tư duy đọc logic code ▾

Không đọc từng dòng rời rạc. Đọc theo chuỗi:

Data → Action → UI Result → Assertion

Với mỗi dòng code, hỏi:

• Dòng này là Setup, Action hay Assertion? • Nó đang gọi method từ đâu? • Data lấy từ file nào? • Expected result là gì? • Nếu fail thì fail do data, state hay locator?
11. Tư duy Assertion Hay yếu ▾

Assertion phải verify behavior, không chỉ verify UI existence.

// ✗ Cách yếu — chỉ biết có row, không biết row đúng không await expect(resultRows.first()).toBeVisible(); // ✓ Cách tốt — verify đúng nội dung const rowCount = await adminPage.resultRows.count(); if (rowCount > 0) { await expect(adminPage.resultRows.first()).toContainText(data.status); } else { await expect(adminPage.resultRows).toHaveCount(0); }
12. Tư duy CI/CD ▾

CI là môi trường hoàn toàn mới. Không có auth/user.json hay browser cache sẵn.

Phải đảm bảo trên CI:

□ Setup chạy trước □ Dependencies được install □ Browser được install □ Session được generate □ Test không phụ thuộc dữ liệu local

CI fail nhưng local pass → kiểm tra environment và setup flow trước.

Viết test case từ POM — từng bước một

Ví dụ thực tế: test chức năng Search User trong trang Admin của OrangeHRM.

1

Tạo Page Object — pages/AdminPage.js

Đây là nơi DUY NHẤT được phép chứa locator. Spec file không được biết locator.

// pages/AdminPage.js class AdminPage { // ─── constructor nhận vào "page" từ Playwright ─────────────────── // "page" là object đại diện cho tab browser đang chạy constructor(page) { this.page = page; // lưu lại để dùng trong các method // ─── LOCATORS ──────────────────────────────────────────────────── // Mỗi dòng này tìm 1 element trên màn hình và gán vào this // Spec sẽ dùng this.xxx thay vì tự tìm element this.usernameInput = page.locator('.oxd-input').nth(1); // └── page.locator('.oxd-input') → tìm tất cả element có class oxd-input // └── .nth(1) → lấy cái thứ 2 (index bắt đầu từ 0) // vì cái đầu tiên (.nth(0)) là input khác this.searchButton = page.getByRole('button', { name: 'Search' }); // └── getByRole('button', ...) → tìm element có role là button // └── { name: 'Search' } → trong đó text hiển thị là "Search" // → Cách này bền hơn locator CSS vì không bị ảnh hưởng khi đổi class this.resultRows = page.locator('.oxd-table-row--clickable'); // └── tìm tất cả row trong bảng kết quả // └── trả về 1 mảng locator, dùng .count() .first() .nth(i) được this.noRecordText = page.getByText('No Records Found'); // └── tìm element có đúng text "No Records Found" // └── Playwright so khớp chính xác text này trên màn hình } // ─── METHODS (hành động người dùng thực hiện) ──────────────────── // Spec sẽ gọi các method này, không cần biết bên trong làm gì async navigate() { // Mở trang Admin User List await this.page.goto('/web/index.php/admin/viewSystemUsers'); } async searchByUsername(username) { // Điền username vào ô tìm kiếm, rồi click Search await this.usernameInput.fill(username); // └── .fill() → xóa nội dung cũ rồi gõ text mới vào await this.searchButton.click(); // └── .click() → bấm vào button } } // Xuất class ra để file khác import được module.exports = { AdminPage };
⚠️ Quy tắc vàng: Nếu bạn thấy mình sắp gõ page.locator() trong file spec → DỪNG LẠI. Đưa cái đó vào POM trước, rồi gọi method từ spec.

Data tách riêng hoàn toàn khỏi logic test. Muốn đổi data → chỉ sửa JSON, không động vào code.

{ "validUser": { "username": "Admin", // └── đây là key bạn sẽ dùng trong spec: data.username "expectedResult": "found" // └── dùng để spec biết nên assert cái gì // └── "found" = kết quả phải hiển thị row }, "invalidUser": { "username": "xyz_not_exist_999", "expectedResult": "notFound" // └── "notFound" = kết quả phải hiển thị "No Records Found" } }

→ Spec code không bị "cứng" với data cụ thể

3

Viết spec file — tests/Admin.spec.js

Spec chỉ làm 3 việc: import POM, import data, gọi method. Không locator, không logic UI.

// tests/Admin.spec.js // ─── IMPORT ────────────────────────────────────────────────────────── const { test, expect } = require('@playwright/test'); // └── test → function để khai báo 1 test case // └── expect → function để viết assertion const { AdminPage } = require('../pages/AdminPage'); // └── import class AdminPage từ POM // └── dùng { } vì AdminPage được export dạng named export const searchData = require('../test-data/adminSearch.json'); // └── import file JSON → tự động parse thành object JavaScript // └── searchData.validUser.username = "Admin" // └── searchData.invalidUser.username = "xyz_not_exist_999" // ─── TEST SUITE ─────────────────────────────────────────────────────── test.describe('Admin - Search User', () => { // └── test.describe() → nhóm nhiều test case lại // └── giúp report dễ đọc hơn, biết test này thuộc nhóm nào let adminPage; // └── khai báo biến ở ngoài để beforeEach và test đều dùng được test.beforeEach(async ({ page }) => { // └── beforeEach chạy TRƯỚC MỖI test case trong describe này // └── { page } → Playwright inject sẵn object page vào đây adminPage = new AdminPage(page); // └── tạo instance của AdminPage, truyền page vào constructor // └── từ đây adminPage.usernameInput, adminPage.searchButton... đều dùng được await adminPage.navigate(); // └── gọi method navigate() trong POM → mở đúng trang Admin // └── await vì navigate() là async (có .goto() bên trong) }); // ─── TEST CASE 1: Positive ────────────────────────────────────────── test('[P] TC_ADMIN_001 - Search valid username returns result', async ({ page }) => { const data = searchData.validUser; // └── lấy object "validUser" từ JSON // └── data.username = "Admin" // └── data.expectedResult = "found" // ACTION await adminPage.searchByUsername(data.username); // └── gọi method trong POM → bên trong POM sẽ: // 1. fill "Admin" vào usernameInput // 2. click searchButton // └── spec KHÔNG cần biết fill hay click cái gì cụ thể // ASSERTION const rowCount = await adminPage.resultRows.count(); // └── kiểm tra có ít nhất 1 row → tức là tìm được kết quả await expect(adminPage.resultRows.first()).toContainText(data.username); // └── .first() → lấy row đầu tiên // └── .toContainText() → kiểm tra row đó chứa text "Admin" // └── dùng data.username thay vì hard-code "Admin" → linh hoạt hơn }); // ─── TEST CASE 2: Negative ────────────────────────────────────────── test('[N] TC_ADMIN_002 - Search invalid username shows no record', async ({ page }) => { const data = searchData.invalidUser; // └── data.username = "xyz_not_exist_999" // ACTION await adminPage.searchByUsername(data.username); // ASSERTION await expect(adminPage.noRecordText).toBeVisible(); // └── adminPage.noRecordText là locator "No Records Found" trong POM // └── .toBeVisible() → kiểm tra element hiển thị trên màn hình }); });

Data Driven Testing — từng bước một

Ví dụ thực tế: test Login với nhiều bộ username/password khác nhau, mỗi bộ cho 1 kết quả khác nhau.

1

Xác định có nên dùng DDT không

Hỏi 3 câu này trước khi code. Nếu cả 3 đều "có" → dùng DDT.

// Câu hỏi 1: Flow có giống nhau không? TC_LOGIN_001: Mở trang → Nhập Admin/admin123 → Click Login → Verify dashboard TC_LOGIN_002: Mở trang → Nhập Wrong/wrong → Click Login → Verify error msg TC_LOGIN_003: Mở trang → Nhập (trống)/admin123 → Click Login → Verify required // └── Cả 3 đều: Mở trang → Nhập data → Click → Verify // └── Flow GIỐNG NHAU ✓ // Câu hỏi 2: Chỉ khác data không? // └── TC1 dùng "Admin" / "admin123" // └── TC2 dùng "Wrong" / "wrong" // └── TC3 dùng "" / "admin123" // └── Chỉ khác ở username và password → KHÁC DATA ✓ // Câu hỏi 3: Expected result có khác nhau không? // └── TC1 expect: URL chứa /dashboard // └── TC2 expect: hiện "Invalid credentials" // └── TC3 expect: hiện "Required" // └── KHÁC EXPECTED RESULT ✓ // ─── KẾT LUẬN: Phù hợp dùng DDT ─────────────────────────────────── // Thay vì viết 3 test case riêng lặp đúng cùng 1 flow, // dùng DDT để viết 1 lần, chạy với 3 bộ data khác nhau
2

Tạo file data — test-data/loginData.json

3

Viết POM — pages/LoginPage.js

⚡ Key Points to Remember

1
Locators are lazy — They don't search until an action is performed. Khai báo locator trong constructor không có gì xảy ra trên browser cả.
2
Auto-waiting — Playwright tự đợi element sẵn sàng trước khi action. Không cần viết sleep() hay waitFor() thủ công.
3
Strict mode — Locator phải match đúng 1 element. Match nhiều hơn → Strict mode violation. Dùng .nth() hoặc narrow locator.
4
Async/Await — Tất cả Playwright function đều async. Phải await mọi thứ, không thì test chạy sai thứ tự.
5
Test isolation — Mỗi test có fresh browser context. State của test này không ảnh hưởng test khác (trừ khi dùng storageState).
6
Built-in assertions — Dùng expect() cho web-specific assertions. Playwright tự retry assertion cho đến timeout, không fail ngay lập tức.
