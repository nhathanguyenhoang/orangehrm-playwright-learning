1. Tư duy tổng quan khi đọc một project Playwright

Khi nhìn vào một project automation, không nên đọc từng file ngẫu nhiên. Nên đọc theo flow tổng thể trước.

playwright.config.js ↓ auth/setup ↓ spec file ↓ page object ↓ test data ↓ report / trace / screenshot
Thành phần Vai trò
playwright.config.jsQuy định cách test chạy
auth.setup.spec.jsChuẩn bị session đăng nhập
*.spec.jsKịch bản test
pages/Chứa locator và action
test-data/Chứa dữ liệu test
report / trace / videoDùng để debug khi fail
Tư duy chính:
Config quyết định cách test chạy. Spec quyết định test làm gì. POM quyết định thao tác trên UI ra sao. Data quyết định input/output. Report cho biết test fail ở đâu.

2. Tư duy cấu trúc code

Một automation project tốt không nên viết tất cả trong một file. Cần tách theo trách nhiệm.

Spec file → mô tả test scenario Page Object → mô tả UI element và action Test Data → mô tả dữ liệu đầu vào và expected result Config → mô tả môi trường chạy test

Ví dụ Admin.spec.js không nên biết locator chi tiết của username input. Nó chỉ nên gọi:

await adminPage.searchByUsername(data.username);

Còn locator thật nên nằm trong AdminPage.js.

Tư duy:
  • Test case nên đọc giống manual test step.
  • Locator nên nằm trong Page Object.
  • Data nên nằm ngoài test logic.

3. Tư duy POM

POM không chỉ để giảm code. POM dùng để gom locator và action về một nơi, giúp spec dễ đọc và dễ bảo trì.

  • Gom locator về một nơi
  • Gom action về một nơi
  • Giúp spec dễ đọc
  • Giúp sửa UI ít ảnh hưởng tới test

Nếu locator của nút Search đổi, chỉ cần sửa trong AdminPage.js, không cần sửa nhiều file test.

Cách đọc POM

constructor() ↓ locator ↓ action method ↓ module.exports
class AdminPage {
  constructor(page) {
    this.page = page;
    this.usernameInput = page.locator('...');
    this.searchButton = page.getByRole('button', { name: 'Search' });
  }

  async searchByUsername(username) {
    await this.usernameInput.fill(username);
    await this.searchButton.click();
  }
}
Locator là “đồ vật trên màn hình”. Method là “hành động của user”. Spec là “kịch bản cần kiểm tra”.

4. Step viết một function trong POM

Khi muốn thêm function mới trong Page Object, làm theo 5 bước.

Bước 1: Xác định mục đích

Tôi muốn search user theo username.

Bước 2: Xác định element cần thao tác

username input
search button

Bước 3: Tạo locator trong constructor

this.usernameInput = page.locator('.oxd-input').nth(1);
this.searchButton = page.getByRole('button', { name: 'Search' });

Bước 4: Viết method hành động

async searchByUsername(username) {
  await this.usernameInput.fill(username);
  await this.searchButton.click();
}

Bước 5: Gọi method trong spec

await adminPage.searchByUsername(data.username);

Checklist khi viết function

  • Function có làm đúng một việc chính không?
  • Tên function có dễ hiểu không?
  • Locator có nằm trong POM không?
  • Spec có bị dính locator không?
  • Function có tái sử dụng được không?

5. Tư duy viết test case

Một test case tốt nên có 3 phần:

Setup Action Assertion

Ví dụ Login thành công

Setup: - Mở trang login Action: - Nhập username - Nhập password - Click Login Assertion: - Kiểm tra URL chứa dashboard
test('[POSITIVE] TC_LOGIN_001 - Verify login succeeds with valid credentials', async ({ page }) => {
  await test.step('Login with valid username and password', async () => {
    await loginPage.performLogin('Admin', 'admin123');
  });

  await test.step('Verify user is redirected to Dashboard page', async () => {
    await expect(page).toHaveURL(/dashboard/);
  });
});
Tư duy: Không viết test chỉ để chạy pass. Viết test để người khác đọc hiểu mục tiêu test.

Một test case nên:

  • Có tên rõ ràng
  • Chỉ kiểm tra một mục tiêu chính
  • Không phụ thuộc test khác
  • Có assertion rõ
  • Có dữ liệu test rõ

6. Tư duy Data Driven Testing

Data-driven chỉ nên dùng khi:

Same flow Different data Different expected result

Ví dụ Login có flow giống nhau:

Nhập username/password Click Login Verify result

Không nên dùng data-driven khi:

  • Flow khác nhau hoàn toàn
  • Assertion khác quá nhiều
  • Phải viết quá nhiều if/else
  • Test trở nên khó đọc hơn
Bài học Sprint 01:
Không phải testcase nào cũng nhét vào loop. Data-driven tốt khi nó làm test gọn hơn. Data-driven tệ khi nó làm logic rối hơn.

7. Tư duy Storage State

Vấn đề ban đầu:

Admin test cần login. Login test lại cần browser chưa login.

Nếu dùng storageState global cho toàn bộ project:

Login test fail vì browser đã login sẵn.

Nếu không dùng storageState:

Admin test bị redirect về login.

Giải pháp đúng:

Auth setup tạo session. Admin test dùng session. Login test không dùng session.
Tư duy quan trọng: Không phải mọi test đều cần cùng một browser state.

Flow đúng

setup project ↓ auth.setup.spec.js ↓ auth/user.json ↓ Admin.spec.js dùng storageState ↓ Login.spec.js chạy browser sạch

8. Tư duy debug Playwright

Khi test fail, không sửa locator ngay. Debug theo thứ tự:

  1. URL hiện tại là gì?
  2. Browser đang login hay chưa login?
  3. storageState có được apply không?
  4. Data input / expected có đúng không?
  5. Locator có tìm đúng element không?
  6. Assertion có đúng business rule không?

Bảng debug nhanh

Hiện tượng Nghi ngờ đầu tiên
Expected /admin/, received /auth/loginSession / storageState lỗi
Locator username không tìm thấyĐang không ở login page
Run riêng pass, run all failConfig / session / shared state
Local pass, CI failEnvironment / session / data / timeout
Có row lúc này, lúc khác khôngDynamic test data
Strict mode violationLocator match nhiều element
Fail message chỉ là triệu chứng. Root cause thường nằm ở flow trước đó.

9. Cách debug Playwright thực tế

Cách 1: Chạy riêng file

npx playwright test tests/Admin.spec.js --headed

Dùng khi muốn biết file đó có tự pass không.

Cách 2: Chạy toàn bộ suite

npx playwright test

Dùng để phát hiện lỗi config, session hoặc shared state.

Cách 3: Mở report

npx playwright show-report

Dùng để xem test fail ở step nào.

Cách 4: Dùng trace/video/screenshot

Khi test fail, xem screenshot, video, trace để biết tại thời điểm fail browser đang ở trang nào.

Cách 5: Dùng error message

Expected: /admin/
Received: /auth/login

Không nên hiểu là assertion sai ngay. Phải hiểu là test muốn vào Admin nhưng bị đá về Login, nghĩa là session không hợp lệ hoặc storageState không được load.

10. Tư duy logic code

Khi đọc một đoạn test code, hỏi:

  • Dòng này thuộc Setup, Action hay Assertion?
  • Nó đang gọi method từ đâu?
  • Data lấy từ file nào?
  • Expected result là gì?
  • Nếu fail thì fail do data, state hay locator?
await adminPage.filterByStatus(data.status);

Đọc logic:

Spec gọi action filterByStatus. Method nằm trong AdminPage. Data status lấy từ adminSearchData.json. Sau đó spec phải verify kết quả filter.
Không đọc code theo từng dòng rời rạc. Đọc theo chuỗi: Data → Action → UI Result → Assertion.

11. Tư duy Assertion

Assertion không nên chỉ kiểm tra “có element”. Phải kiểm tra đúng mục tiêu test.

Cách yếu

await expect(resultRows.first()).toBeVisible();

Cách này chỉ biết có row, không biết row có đúng status không.

Cách tốt hơn

const rowCount = await adminPage.resultRows.count();

if (rowCount > 0) {
  await expect(adminPage.resultRows.first()).toContainText(data.status);
} else {
  await expect(adminPage.resultRows).toHaveCount(0);
}
Tư duy: Verify behavior, not just UI existence.

12. Tư duy CI/CD

CI không giống local.

Local có thể đã có:

auth/user.json browser cache network ổn máy mạnh

CI là môi trường mới. Vì vậy phải đảm bảo:

  • Setup chạy trước
  • Dependencies được install
  • Browser được install
  • Session được generate
  • Test không phụ thuộc dữ liệu local
Nếu CI fail nhưng local pass, hãy kiểm tra environment và setup flow trước.

16. Git Workflow Summary

Trong Sprint 01, mình không chỉ viết test mà còn thực hành workflow gần giống môi trường đi làm.

Flow Git đã thực hành

Issue ↓ Create branch ↓ Code change ↓ Run test local ↓ Commit ↓ Push ↓ Pull Request ↓ GitHub Actions ↓ Merge ↓ Revert nếu cần ↓ Fix bằng issue/branch mới

Các command đã dùng

git branch
git checkout -b feature/auth-storage-state
git branch -m feature/auth-storage-state
git add .
git commit -m "Implement storage state authentication and admin tests"
git push -u origin feature/auth-storage-state
git fetch origin
git checkout main
git pull origin main
git log --oneline --decorate -10

Bài học Git quan trọng

  • branch dùng để làm việc riêng, không sửa trực tiếp trên main.
  • commit là một mốc thay đổi có ý nghĩa.
  • Pull Request dùng để review và merge code vào main.
  • Revert không xóa lịch sử, mà tạo commit mới để đảo ngược thay đổi cũ.
  • Nếu PR bị revert, cách sạch hơn là tạo issue/branch mới rồi fix lại.
Tư duy đi làm: Không chỉ code cho chạy pass. Phải biết đưa code lên branch, tạo PR, đọc CI, xử lý fail và rollback khi cần.

19. Câu hỏi tự ôn sau Sprint 01

Phần này dùng để tự học, tự grill nhẹ hoặc chuẩn bị phỏng vấn QA Automation.

1. POM dùng để làm gì?

Trả lời: POM dùng để tách locator và action ra khỏi spec file. Spec chỉ mô tả test scenario, còn Page Object chịu trách nhiệm thao tác với UI. Khi UI thay đổi, mình chỉ cần sửa locator trong Page Object thay vì sửa nhiều file test.

2. Khi nào nên dùng data-driven testing?

Trả lời: Khi nhiều test case có cùng flow nhưng khác input và expected result. Ví dụ login với valid user, invalid password, invalid username đều có cùng flow nhập username/password rồi click Login.

3. Khi nào không nên dùng data-driven?

Trả lời: Không nên dùng khi các case có flow quá khác nhau hoặc phải viết quá nhiều if/else. Lúc đó test sẽ khó đọc và khó maintain hơn.

4. Vì sao không apply storageState global cho toàn project?

Trả lời: Vì không phải test nào cũng cần trạng thái đã login. Admin test cần session đã login, nhưng Login test cần browser sạch để kiểm tra login thật. Nếu apply global, Login test có thể fail vì browser đã login sẵn.

5. Vì sao dùng Playwright project dependencies?

Trả lời: Để auth setup chạy trước và tạo auth/user.json, sau đó các test cần authenticated session mới dùng file đó. Cách này giúp CI ổn định hơn và tránh việc setup chạy lẫn với test thường.

6. Khi thấy Expected /admin/ nhưng Received /auth/login, nên nghĩ gì đầu tiên?

Trả lời: Nên nghi ngờ session hoặc storageState. Test muốn vào Admin nhưng bị đá về Login nghĩa là browser chưa có session hợp lệ hoặc chưa load đúng auth/user.json.

7. Vì sao Run riêng PASS nhưng Run All FAIL?

Trả lời: Thường do config, shared state, session, test data hoặc test phụ thuộc nhau. Không nên sửa locator ngay. Nên check config và state trước.

8. Vì sao Disabled status filter không nên expect cứng là có row hoặc No Records Found?

Trả lời: Vì dữ liệu demo có thể thay đổi. Hôm nay không có disabled user, ngày mai có thể có. Test nên verify business rule: nếu có row thì row phải đúng status, nếu không có row thì table count bằng 0.

9. Một test case tốt nên có cấu trúc gì?

Trả lời: Nên có Setup, Action và Assertion. Test name phải rõ mục tiêu, test không phụ thuộc case khác và assertion phải kiểm tra đúng behavior cần test.

10. Khi CI fail nhưng local pass, nên kiểm tra gì?

Trả lời: Kiểm tra setup flow, dependencies, browser install, session generation, environment data, timeout và GitHub Actions logs. CI là môi trường sạch nên không thể phụ thuộc file hoặc cache local.
